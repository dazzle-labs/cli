## 1. Text-to-Speech for Narration

The agent stream needs a voice. The agent is doing real work (fetching data, analyzing situations, composing visual output) and narrating that work makes it watchable. TTS is the mechanism that turns the agent's reasoning into speech the viewer hears.

### 1a. ElevenLabs

**Quality**: Best-in-class voice quality as of early 2026. Natural prosody, emotional range, minimal artifacts. Widely regarded as the gold standard for realistic TTS. Expressive enough to convey urgency, calm, and curiosity without sounding robotic.

**Latency**: Streaming API delivers first audio chunk in ~300-500ms time-to-first-byte. Their Turbo v2.5 model targets sub-300ms. Full streaming mode sends audio chunks as they are generated, so playback begins before the full audio is synthesized. Acceptable for live narration.

**Streaming API** (WebSocket, recommended for live streams):

```typescript
import { ElevenLabsClient } from '@elevenlabs/elevenlabs-js';

const client = new ElevenLabsClient({ apiKey: process.env.ELEVENLABS_API_KEY });

async function streamTTS(text: string, voiceId: string): Promise<ReadableStream> {
  const audioStream = await client.textToSpeech.streamWithTimestamps(voiceId, {
    text,
    modelId: 'eleven_turbo_v2_5',  // Lowest latency model
    outputFormat: 'pcm_24000',      // Raw PCM, ideal for Web Audio API
    voiceSettings: {
      stability: 0.5,
      similarityBoost: 0.75,
    },
  });
  return audioStream;
}
```

**Chunked streaming** (send partial sentences as they are generated by LLM):

```typescript
// Stream text from LLM and pipe directly to ElevenLabs
const textStream = streamLLMOutput(prompt);
const audioStream = client.textToSpeech.streamInput(voiceId, {
  modelId: 'eleven_turbo_v2_5',
  outputFormat: 'pcm_24000',
});

// As each LLM token arrives, send to TTS
for await (const chunk of textStream) {
  audioStream.send(chunk);
}
audioStream.flush();
```

This pipeline (LLM streaming -> TTS streaming) can reduce latency from text input to first audible speech to under 500ms, because TTS starts generating audio before the LLM has finished generating text.

**Voices**: 30+ premade voices, professional voice cloning (from ~1 minute of sample audio), voice design. Can create a consistent "Dazzle anchor" voice that remains stable across all narration.

**Formats**: `mp3_44100_128`, `pcm_16000`, `pcm_22050`, `pcm_24000`, `pcm_44100`, `ulaw_8000`. Use PCM for lowest latency (no decode overhead).

**Pricing** (as of early 2026):

|------|---------|---------------|------------|
| Free | $0 | 10,000 | -- |
| Starter | $5 | 30,000 | $0.167 |
| Creator | $22 | 100,000 | $0.220 |
| Pro | $99 | 500,000 | $0.198 |
| Scale | $330 | 2,000,000 | $0.165 |

Usage API (pay-as-you-go): approximately $0.18-0.30 per 1,000 characters depending on model.

**Cost per hour of speech**: Average speaking rate is ~150 words/minute = 9,000 words/hour = ~45,000 characters/hour. At the Scale tier, $0.165/1K chars = **$7.43/hr of continuous speech**. At usage pricing (~$0.24/1K), **$10.80/hr**.

**Key consideration**: ElevenLabs is expensive for continuous narration. Best used selectively for high-impact moments or for pre-generating a branded intro/outro voice that gets cached and reused.

### 1b. OpenAI TTS

**Quality**: Very good. Three models: `tts-1` (speed-optimized), `tts-1-hd` (quality), and `gpt-4o-mini-tts` (instruction-following). The `gpt-4o-mini-tts` model lets you prepend instructions like "Speak with urgency, like a news anchor delivering breaking news" or "Speak calmly and analytically" -- directly in the API call.

**Latency**: `tts-1` targeting ~400-600ms time-to-first-byte with streaming. `gpt-4o-mini-tts` similar. `tts-1-hd` is slower (~1-2s, not suitable for live narration).

**REST Streaming API**:

```typescript
const openai = new OpenAI();

async function streamTTS(text: string, instructions?: string): Promise<Buffer> {
  const response = await openai.audio.speech.create({
    model: 'gpt-4o-mini-tts',
    voice: 'nova',
    input: text,
    // Instruction-following: control tone dynamically
    ...(instructions && { voice_instructions: instructions }),
    response_format: 'pcm',  // Raw PCM, no container overhead
    speed: 1.0,
  });

  // Stream audio chunks as they arrive
  const stream = response.body;
  // Pipe to Web Audio API decoder in Chrome via WebSocket
  return streamToWebSocket(stream);
}

// Example usages with dynamic tone control:
await streamTTS(
  'Breaking development: new sanctions announced targeting Iranian oil exports.',
  'Speak with urgency and gravitas, like a news anchor delivering breaking news.'
);

await streamTTS(
  'Fetching the latest satellite imagery. Processing complete. Three new convoy positions identified.',
  'Speak in a measured, analytical tone, like an intelligence analyst briefing.'
);
```

**Voices**: 6 built-in voices. `nova` (warm, engaging), `onyx` (deep, authoritative), `alloy` (neutral), `echo` (crisp), `fable` (expressive), `shimmer` (soft).

**Pricing**:

| Model | Cost per 1M characters |
|-------|------------------------|
| `tts-1` | $15.00 |
| `tts-1-hd` | $30.00 |
| `gpt-4o-mini-tts` | $12.00 |

**Cost per hour of continuous speech** (45,000 chars):

| Model | Cost/hr |
|-------|---------|
| `tts-1` | $0.675 |
| `tts-1-hd` | $1.35 |
| `gpt-4o-mini-tts` | $0.540 |

**Key consideration**: Best cost-to-quality ratio for agent streams. The `gpt-4o-mini-tts` instruction-following feature is uniquely powerful -- the agent's LLM can generate narration text *and* specify how it should be spoken, without changing voice IDs. This makes it possible to have a single consistent narrator that sounds different for breaking news vs. routine updates.

```typescript
    streamingConfig: {

      audioConfig: {

        sampleRateHertz: 24000,

}
```

### 1d. Microsoft Azure / Edge TTS

**Quality**: Azure Neural voices are competitive with Google Neural2. Some Azure voices have excellent natural prosody. Edge TTS specifically refers to the voices available via the browser's `SpeechSynthesis` API backed by Microsoft's neural TTS infrastructure.

**Latency**: Azure Cognitive Services TTS streaming typically delivers first audio in ~300-500ms. The `edge-tts` open-source library accesses the Azure TTS endpoint used by Microsoft Edge's read-aloud feature, which has similar latency.

**Edge TTS (Free, unofficial)**:

```bash

# via the same endpoint used by Microsoft Edge browser
pip install edge-tts
edge-tts --voice en-US-AriaNeural --text "Breaking news alert." --write-media output.mp3
```

```typescript
// Node.js usage via edge-tts package

const tts = new EdgeTTS();
await tts.tts('Breaking news alert.', 'en-US-AriaNeural');
const audioBuffer = await tts.toBuffer();
```

Edge TTS provides access to ~300 neural voices across 70+ languages for free by using the same API endpoint that powers Microsoft Edge's built-in read-aloud feature. This is technically unofficial use and could be rate-limited or discontinued.

**Azure Cognitive Services TTS (Official)**:

```typescript
import * as sdk from 'microsoft-cognitiveservices-speech-sdk';

const speechConfig = sdk.SpeechConfig.fromSubscription(process.env.AZURE_SPEECH_KEY!, 'eastus');
speechConfig.speechSynthesisVoiceName = 'en-US-AriaNeural';
speechConfig.speechSynthesisOutputFormat = sdk.SpeechSynthesisOutputFormat.Raw24Khz16BitMonoPcm;

const synthesizer = new sdk.SpeechSynthesizer(speechConfig);

function streamNarration(text: string): Promise<ArrayBuffer> {
  return new Promise((resolve, reject) => {
    synthesizer.speakTextAsync(
      text,
      result => resolve(result.audioData),
      error => reject(error)
    );
  });
}
```

**Azure Pricing**: ~$16/1M characters for Neural voices (same as Google Neural2), with 500K free chars/month.

**Key consideration**: Edge TTS (free, unofficial) works well for development and low-volume use cases. For production, Azure Neural pricing matches Google. Neither has the instruction-following capability of OpenAI `gpt-4o-mini-tts`.

### 1e. Amazon Polly

**Latency**: Very fast. First bytes in ~100-300ms. Polly was built for real-time and it shows. Lowest-latency option of all the major providers.

**Streaming API**:

```typescript
import { PollyClient, SynthesizeSpeechCommand } from '@aws-sdk/client-polly';

const polly = new PollyClient({ region: 'us-east-1' });

async function streamTTS(text: string): Promise<ReadableStream> {

    SampleRate: '24000',

}
```

### 1h. Open Source: Coqui, Bark, XTTS

#### Coqui TTS / XTTS-v2

**What it is**: Coqui TTS is an open-source TTS framework. XTTS-v2 is their flagship model, capable of high-quality voice cloning from as little as 3 seconds of reference audio.

**Quality**: Surprisingly good for an open-source model. XTTS-v2 produces natural-sounding speech that is competitive with commercial Neural2/Polly quality. Not quite ElevenLabs, but within range.

**Latency**: Depends on hardware. On a CPU, synthesis is slower-than-real-time for long text. On a GPU (e.g., a T4 or A10), XTTS-v2 can synthesize faster than real-time for short to medium segments. Typical first-audio latency on GPU: 500ms-2s for a sentence.

**Streaming support**: Not native in the base model. The output is typically a full audio buffer. Some implementations wrap it with chunked generation.

**Self-hosting**:

```python
# Coqui XTTS-v2 inference
from TTS.api import TTS

tts = TTS("tts_models/multilingual/multi-dataset/xtts_v2").to("cuda")

# Voice cloning from reference audio
tts.tts_to_file(
    text="The situation monitor has identified three new developments.",
    speaker_wav="reference_voice.wav",  # 3+ seconds of reference audio
    language="en",
    file_path="output.wav"
)
```

**Voice cloning**: Can create a consistent "Dazzle narrator" voice from ~30 seconds of reference audio. The voice is then available for all narration at zero marginal cost.

**Cost**: Zero per-character cost after hardware. Running on a shared GPU alongside video generation could work if generation is not always saturating the GPU. On a dedicated CPU instance, synthesis speed may be insufficient for real-time without queuing.

**Key consideration**: XTTS-v2 is viable if you want zero marginal TTS cost and have GPU headroom. The latency (500ms-2s) is manageable if narration is queued and played slightly delayed rather than requiring immediate response. For pre-recorded announcements and cached narration, excellent choice.

#### Bark (Suno's Open Source TTS)

**What it is**: Bark is a transformer-based text-to-audio model from Suno that generates not just speech but also music, background sounds, and nonverbal communications (laughing, sighing, etc.).

**Quality**: Highly realistic, very expressive. Can generate laughter, singing, non-verbal sounds. Output quality is striking but inconsistent -- sometimes the output is excellent, sometimes garbled.

**Latency**: Very slow. Bark generates audio from a transformer that processes the whole sequence. A single sentence takes 5-30 seconds on a GPU. Not suitable for real-time streaming.

**Interesting capabilities**:

```python
from bark import generate_audio, preload_models
import soundfile as sf

preload_models()

# Generate speech with emotion tags
text_prompt = "[excited] Breaking development! [sighs] Though the data suggests... [clears throat] Let me summarize what this means."
audio_array = generate_audio(text_prompt)
sf.write("output.wav", audio_array, 24000)
```

**Key consideration**: Bark is not viable for real-time streaming due to generation speed. Use it for pre-generating unique, expressive audio clips that get cached (intros, special moments, branded sound elements). The nonverbal communication capability (laughter, sighs) could make pre-generated content feel more human.

#### StyleTTS 2 / NaturalSpeech 3

**What they are**: Research-grade open-source TTS models with state-of-the-art quality scores. Less production-ready than Coqui but impressive quality.

**Practical assessment**: These require significant engineering to deploy and run reliably. Use only if you have a dedicated ML engineer and a strong reason to avoid commercial TTS APIs.

#### Piper TTS

**What it is**: A lightweight, fast neural TTS specifically designed to run on CPU in real-time. Used in Home Assistant and other embedded/edge applications.

**Quality**: Below Neural2/OpenAI tts-1, but surprisingly good for a CPU model. Sounds more natural than the old Festival/eSpeak-style TTS.

**Latency**: Designed for real-time CPU inference. First audio typically in ~100-200ms on modern hardware.

**Cost**: Zero. Runs entirely locally.

**Key consideration**: Best zero-cost, real-time TTS option. Worth keeping as a fallback if API-based TTS fails or the budget is exhausted.

```bash
# Install and run Piper TTS
pip install piper-tts
echo "The monitoring agent has identified three new developments." | \
  piper --model en_US-amy-medium --output_file output.wav
```

### 1i. TTS Latency Analysis for Live Streams

Latency budget for agent narration (time from agent deciding to narrate to viewer hearing audio):

```
LLM generates narration text:     200-500ms   (streaming from LLM)
TTS first audio chunk:             250-600ms   (provider-dependent)
WebSocket to React app:            1-5ms       (local network)
Web Audio API decode + buffer:     5-20ms      (Chrome)
PulseAudio capture:                ~0ms        (virtual sink)
FFmpeg encode + RTMP:              33-66ms     (1-2 frames at 30fps)
CDN ingest to viewer:              2-15s       (Twitch/YouTube)
-----------------------------------------------------------------
Total (agent decision to viewer):  2.5-16s
Total (agent decision to Chrome):  456ms-1.1s  (what we control)
```

The CDN latency (2-15 seconds) dominates. The TTS provider choice affects local latency (within Chrome), not the end-to-end viewer experience. This means:

1. **TTS latency matters for synchronization** (keeping audio in sync with visual content in the React app), not for reducing what viewers experience as lag.
2. **The real constraint is** making sure TTS audio starts playing in Chrome's Web Audio API before the corresponding visual moment appears on screen.
3. **Strategy**: Schedule narration slightly ahead of the visual it accompanies. If a data card animates in at T=0, start the TTS audio at T=-500ms so Chrome plays it just as the visual appears.

## 2. Background Music and Ambient Audio

### 2a. ACE-Step 1.5

ACE-Step 1.5 is an open-source music generation model mentioned in the project's technical direction. It generates high-quality music from text prompts.

**What it is**: A step-based audio generation model (related to the "step" diffusion paradigm) capable of generating diverse music styles from text descriptions. Open-source, available for self-hosting.

**Generation quality**: Competitive with Suno and Udio for instrumental music. Can generate looping ambient tracks, stingers, and beds appropriate for news/information streams.

**Generation speed**: Not real-time. A 30-second ambient loop takes 10-60 seconds to generate depending on hardware. This is fine for pre-generating a library, not for on-demand mid-stream generation.

**Self-hosting**:

```bash
# Clone and run ACE-Step
git clone https://github.com/ace-step/ACE-Step
cd ACE-Step
pip install -r requirements.txt

# Generate a music track
python inference.py \
  --prompt "calm ambient news monitoring background music, subtle electronics, professional" \
  --duration 120 \
  --output ambient-news-bed.wav
```

**Strategic use**: Pre-generate a library of 50-100 background tracks categorized by mood. This is a one-time operation. The GPU time to generate 100 tracks at ~30 seconds each is approximately 50-100 minutes of GPU time, costing $2-5 at serverless GPU rates. After that, the library costs nothing.

### 2e. Web Audio API for Synthesized Backgrounds

The browser can generate ambient audio entirely in code using the Web Audio API, with zero cost and zero latency:

```typescript
// Generative ambient background synthesizer
class AmbientSynthesizer {
  private ctx: AudioContext;
  private masterGain: GainNode;

  constructor(ctx: AudioContext) {
    this.ctx = ctx;
    this.masterGain = ctx.createGain();
    this.masterGain.gain.value = 0.05;  // Very quiet background
    this.masterGain.connect(ctx.destination);
  }

  // Low-frequency drone using oscillators
  startDrone(frequency: number = 55): void {
    const osc1 = this.ctx.createOscillator();
    const osc2 = this.ctx.createOscillator();
    const filter = this.ctx.createBiquadFilter();

    osc1.type = 'sine';
    osc1.frequency.value = frequency;
    osc2.type = 'sine';
    osc2.frequency.value = frequency * 1.001;  // Slight detune for warmth

    filter.type = 'lowpass';
    filter.frequency.value = 300;  // Remove harshness

    osc1.connect(filter);
    osc2.connect(filter);
    filter.connect(this.masterGain);

    osc1.start();
    osc2.start();
  }

  // Periodic soft pings (like a radar)
  startPingPattern(interval: number = 4000): void {
    const ping = () => {
      const osc = this.ctx.createOscillator();
      const env = this.ctx.createGain();
      osc.frequency.value = 880;
      osc.connect(env);
      env.connect(this.masterGain);
      env.gain.setValueAtTime(0, this.ctx.currentTime);
      env.gain.linearRampToValueAtTime(0.1, this.ctx.currentTime + 0.01);
      env.gain.exponentialRampToValueAtTime(0.001, this.ctx.currentTime + 0.5);
      osc.start();
      osc.stop(this.ctx.currentTime + 0.5);
      setTimeout(ping, interval);
    };
    ping();
  }
}
```

**Value of synthesized audio**: Zero cost, always on, no file storage needed. Creates a distinctive "live monitoring" feel. Works well for data-monitoring streams where a slight technological hum adds to the atmosphere.

**Limitation**: Synthesized audio sounds obviously electronic. Better for tech/data stream aesthetic than news/journalism aesthetic.

## 3. Sound Design for Agent Streams

### 3a. How Broadcast News Handles Audio Transitions

**CNN audio design principles**:
- Iconic "breaking news" alert: ~1.5 second urgent brass stinger
- Background music beds during non-breaking coverage: -25dB to -35dB below spoken word volume, so barely perceptible
- Transition sounds between segments: short whoosh or tonal hit (~0.3s)
- Lower-third animations often have subtle sound effects

**Bloomberg TV audio design**:
- Minimal background music; often just silence or a very low pad
- Alert sounds are short, clean, distinctive (2-3 note motifs)
- Breaking news has an escalating tone sequence
- Data updates are visual-only; audio reserved for human anchors
- The design is intentionally sparse to prevent fatigue across a 24/7 channel

**BBC World Service radio principles** (highly relevant for information streams):
- Silence is used deliberately and not filled
- Each segment type has a distinctive audio signature
- Sound design is predictable and consistent -- viewers learn what sounds mean
- Music beds are mixed very low (under -20dB) during speech
- Stingers (transition sounds) are always the same length within a show

**Key principle**: Audio design that works for 5 minutes can be deeply fatiguing at 5 hours. Agent streams may run continuously. The design must be livable at scale.

---

### 3b. Sound Design Taxonomy for Agent Streams

**Event-to-sound mapping**:

| Agent Event | Sound Character | Duration | Relative Volume |
|------------|----------------|----------|-----------------|
| Stream started | Opening theme sting | 5-8s | High (branded) |
| New data source connected | Soft ascending chime | 0.5s | Low |
| Data fetch complete | Single soft click | 0.1s | Very low |
| Significant finding identified | Two-note ascending motif | 0.8s | Medium |
| Breaking / urgent development | Three-note urgent brass stab | 1.2s | High |
| Task completed successfully | Resolved ascending chord | 0.6s | Low |
| Error or issue encountered | Descending minor tone | 0.8s | Medium |
| Topic transition | Whoosh + light sting | 0.4s | Low |
| Agent speaks (TTS start) | Optional very subtle lead-in | 0.05s | Minimal |
| Stream paused | Fade out + silence | 1.5s | Decreasing |
| Stream resuming | Fade in ambient | 1.5s | Increasing |

**Anti-patterns** (sounds that drive viewers away):

- Playing a sound for every single data fetch (streams can fetch 50-100 sources/hr)
- High-pitched notification sounds (fatiguing in minutes)
- Variable timing for the same event (unpredictability creates anxiety)
- Loud music competing with narration
- Total silence for 30+ seconds without intentional design (signals broken stream)
- Sound effects that are culturally associated with failure/error when the event is neutral

### 3c. Sonification of Agent Activity

**Data sonification** is the practice of encoding data dimensions into audio parameters. For agent streams, this can create a useful ambient layer of "monitoring is happening":

**Typing/keyboard sounds for coding streams**:

```typescript
// Generate realistic keyboard sounds using Web Audio API
class KeyboardSonifier {
  private ctx: AudioContext;

  // Called each time agent writes a line of code
  onCodeWrite(linesPerMinute: number): void {
    // Faster lines = faster typing sounds
    const charRate = linesPerMinute * 40;  // ~40 chars per line
    this.scheduleTypingBurst(charRate, 2000);  // 2 second burst
  }

  private scheduleTypingBurst(charsPerSec: number, duration: number): void {
    const interval = 1000 / charsPerSec;
    let elapsed = 0;
    while (elapsed < duration) {
      const t = this.ctx.currentTime + elapsed / 1000;
      this.playKeystroke(t);
      elapsed += interval + (Math.random() - 0.5) * interval * 0.3;  // Humanize timing
    }
  }

  private playKeystroke(when: number): void {
    // Noise burst through bandpass filter -- sounds like a keyswitch
    const bufferSize = this.ctx.sampleRate * 0.02;  // 20ms
    const buffer = this.ctx.createBuffer(1, bufferSize, this.ctx.sampleRate);
    const data = buffer.getChannelData(0);
    for (let i = 0; i < bufferSize; i++) {
      data[i] = (Math.random() * 2 - 1) * Math.exp(-i / (bufferSize * 0.3));
    }

    const source = this.ctx.createBufferSource();
    const filter = this.ctx.createBiquadFilter();
    const gain = this.ctx.createGain();

    source.buffer = buffer;
    filter.type = 'bandpass';
    filter.frequency.value = 2000 + Math.random() * 1000;  // Slight frequency variation
    filter.Q.value = 2;
    gain.gain.value = 0.02;  // Very quiet

    source.connect(filter).connect(gain).connect(this.ctx.destination);
    source.start(when);
  }
}
```

## 4. Technical Implementation in Headless Chrome

### 4a. Web Audio API in Headless Mode

**Does Web Audio API work in headless Chrome?** Yes, with one requirement:

```bash
# Chrome must allow autoplay without user gesture
google-chrome \
  --headless=new \
  --autoplay-policy=no-user-gesture-required \
  --disable-gpu \
  --no-sandbox \
  http://localhost:3000
```

The `--autoplay-policy=no-user-gesture-required` flag is essential. Without it, Chrome blocks audio from starting automatically (a security measure for headed browsers). In headless/server mode, this restriction must be removed.

**AudioContext state**: Even with the flag set, the AudioContext may start in `suspended` state. The React app must call `ctx.resume()` programmatically:

```typescript
useEffect(() => {
  const ctx = new AudioContext();
  // Resume immediately -- in headless Chrome, this should succeed without user gesture
  ctx.resume().then(() => {
    console.log('AudioContext state:', ctx.state);  // Should be 'running'
    initAudioMixer(ctx);
  });
}, []);
```

### 4c. Mixing Multiple Audio Sources

The Web Audio API is the right place to mix sources, not FFmpeg. The React app becomes the audio mixer:

```typescript
class StreamAudioMixer {
  private ctx: AudioContext;
  private masterCompressor: DynamicsCompressorNode;

  // Separate gain nodes for each channel
  private ttsGain: GainNode;
  private musicGain: GainNode;
  private sfxGain: GainNode;
  private synth: GainNode;

  constructor() {
    this.ctx = new AudioContext({ sampleRate: 44100 });

    // Master compressor: prevents clipping when multiple sources peak simultaneously
    this.masterCompressor = this.ctx.createDynamicsCompressor();
    this.masterCompressor.threshold.value = -6;    // dB; start compressing here
    this.masterCompressor.knee.value = 3;
    this.masterCompressor.ratio.value = 4;
    this.masterCompressor.attack.value = 0.003;    // 3ms attack
    this.masterCompressor.release.value = 0.25;   // 250ms release
    this.masterCompressor.connect(this.ctx.destination);

    // TTS: full volume speech
    this.ttsGain = this.ctx.createGain();
    this.ttsGain.gain.value = 1.0;
    this.ttsGain.connect(this.masterCompressor);

    // Music: quiet background
    this.musicGain = this.ctx.createGain();
    this.musicGain.gain.value = 0.07;  // ~-23dB relative to speech
    this.musicGain.connect(this.masterCompressor);

    // SFX: medium volume, brief
    this.sfxGain = this.ctx.createGain();
    this.sfxGain.gain.value = 0.25;
    this.sfxGain.connect(this.masterCompressor);

    // Synthesized ambient: very quiet
    this.synth = this.ctx.createGain();
    this.synth.gain.value = 0.04;
    this.synth.connect(this.masterCompressor);
  }

  // Play TTS audio with automatic music ducking
  async playTTS(audioBuffer: ArrayBuffer): Promise<void> {
    const buffer = await this.ctx.decodeAudioData(audioBuffer);
    const source = this.ctx.createBufferSource();
    source.buffer = buffer;
    source.connect(this.ttsGain);

    // Duck music during TTS
    this.musicGain.gain.linearRampToValueAtTime(0.03, this.ctx.currentTime + 0.2);

    source.start();
    source.onended = () => {
      // Restore music after TTS ends
      this.musicGain.gain.linearRampToValueAtTime(0.07, this.ctx.currentTime + 0.5);
    };
  }

  // Stream TTS audio chunks via PCM
  playTTSStream(pcmChunks: AsyncIterable<ArrayBuffer>): void {
    // Queue PCM chunks and play them in sequence
    // Using ScriptProcessorNode or AudioWorklet for low-latency streaming playback
    const worklet = new AudioWorkletNode(this.ctx, 'streaming-pcm-player');
    worklet.connect(this.ttsGain);

    (async () => {
      for await (const chunk of pcmChunks) {
        worklet.port.postMessage({ chunk });
      }
    })();
  }

  // Set background music track
  setMusicTrack(src: string): void {
    const audio = new Audio(src);
    audio.loop = true;
    const source = this.ctx.createMediaElementSource(audio);
    source.connect(this.musicGain);
    audio.play();
  }

  // Play a sound effect
  playSFX(sound: 'chime' | 'alert' | 'transition' | 'resolved' | 'error'): void {
    const audio = new Audio(`/assets/audio/sfx/${sound}.wav`);
    const source = this.ctx.createMediaElementSource(audio);
    source.connect(this.sfxGain);
    audio.play();
  }
}
```

### 4d. Streaming TTS PCM to Web Audio API

Getting low-latency TTS audio into the browser's Web Audio API requires handling PCM streaming:

```typescript
// AudioWorklet processor for streaming PCM playback
// (Define in a separate file: streaming-pcm-player.worklet.js)
class StreamingPCMPlayer extends AudioWorkletProcessor {
  private buffer: Float32Array[] = [];
  private position = 0;

  constructor() {
    super();
    this.port.onmessage = (e) => {
      if (e.data.chunk) {
        // Convert incoming Int16 PCM to Float32
        const int16 = new Int16Array(e.data.chunk);
        const float32 = new Float32Array(int16.length);
        for (let i = 0; i < int16.length; i++) {
          float32[i] = int16[i] / 32768;
        }
        this.buffer.push(float32);
      }
    };
  }

  process(inputs: Float32Array[][], outputs: Float32Array[][]): boolean {
    const output = outputs[0][0];
    let outputPos = 0;

    while (outputPos < output.length && this.buffer.length > 0) {
      const chunk = this.buffer[0];
      const available = chunk.length - this.position;
      const needed = output.length - outputPos;
      const toCopy = Math.min(available, needed);

      output.set(chunk.subarray(this.position, this.position + toCopy), outputPos);
      this.position += toCopy;
      outputPos += toCopy;

      if (this.position >= chunk.length) {
        this.buffer.shift();
        this.position = 0;
      }
    }

    return true;  // Keep processor alive
  }
}

registerProcessor('streaming-pcm-player', StreamingPCMPlayer);
```

```typescript

  }
}
```

### 4e. Audio Latency and Synchronization

The key synchronization problem: **TTS audio must align with the visual content it describes**.

If the agent narrates "Here's the latest data from Tehran" and simultaneously displays a map of Tehran, the audio and visual must arrive together. In a live stream with 5-15 seconds of CDN latency, both audio and video have the same delay. The sync problem exists only within Chrome, before capture.

**Sync strategy**:

```typescript
class AudioVideoSync {
  // When agent decides to show content AND narrate:
  async showAndNarrate(
    visualContent: ContentUpdate,
    narrationText: string
  ): Promise<void> {
    // Start both simultaneously
    const [audioPromise, visualPromise] = await Promise.all([
      this.prepareTTSAudio(narrationText),    // Pre-fetch audio
      this.prepareVisualUpdate(visualContent),  // Pre-render component
    ]);

    // Apply both at the same moment
    const t = audioCtx.currentTime + 0.05;  // 50ms in the future
    this.scheduleAudio(audioPromise, t);
    this.scheduleVisual(visualPromise, t);
  }

  // For time-sensitive narration (agent discovers something while stream is live):
  async narrateImmediate(text: string): Promise<void> {
    // Fire and forget; narration aligns with current visual
    // Acceptable slight async offset; viewers cannot detect sub-500ms offset
    this.playTTSStream(text);
  }
}
```

## 6. What Makes Agent Stream Audio Interesting

### 6a. Coding Streams

A coding agent stream (e.g., an agent working through a GitHub repository) has natural sonic opportunities:

**Ambient layer (always on):**
- Subtle keyboard sounds synced to when the agent writes code (typing rate proportional to actual LOC/min)
- Low processing hum that intensifies when compilation happens
- The sound of a "working" environment -- different from silence

**Event sounds:**
- Test pass: single ascending two-note chime
- Test fail: descending minor tone (firm but not alarming)
- Build success: resolved chord, slightly triumphant
- Build failure: two descending tones, suggests "try again"
- First commit pushed: brief celebratory chord
- PR opened: chime + brief sting
- PR merged: celebratory resolution

**Narration opportunities for coding streams:**
- "I've identified a bug in the authentication handler. Let me trace through the call stack."
- "Running the test suite now. 47 tests passing, 2 failures I need to address."
- "This function is doing too much. I'm going to refactor it into three smaller functions."

The narration should sound like a developer explaining their work to a colleague. Not robotic, not scripted. The instruction-following in OpenAI's `gpt-4o-mini-tts` is valuable here: "speak like a thoughtful developer talking through a problem."

**What makes it engaging**: The viewer can hear the agent is *actually working*. The rhythm of typing sounds, build sounds, and selective narration creates a sense of authentic activity. Compare to watching someone code on Twitch -- the appeal is the authenticity of watching real work.

**What makes it annoying**: Narrating every single line ("Now I'm adding a semicolon. Now I'm defining a variable."). Loud keyboard sounds. Frequent loud test-fail sounds on a flaky test suite.

---

### 6b. Data / Situation Monitoring Streams

The Iran situation monitor or similar data-focused streams:

**Ambient layer:**
- Low pulsing tone (radar/sonar aesthetic) that suggests active monitoring
- Very subtle data-fetch clicks on each API call (batched, not per-call)
- A sense of constant active processing

**Event sounds:**
- New data source connected: soft ascending chime
- Significant development identified: news sting (2-note brass)
- Breaking/urgent: three-note brass stab (CNN-style)
- Regular update cycle: nearly inaudible soft click
- Shift to new topic area: transition whoosh

**Narration opportunities:**
- "Monitoring systems are tracking increased naval activity in the Strait of Hormuz."
- "Cross-referencing three separate satellite imagery sources. Analysis complete: convoy movement identified."
- "Breaking: new sanctions package announced. Switching to full coverage mode."

**What makes it engaging**: The audio communicates the urgency hierarchy clearly. Viewers can listen with half attention (like background radio) and know when to pay closer attention based on audio cues. The ambient monitoring sounds create the sense of a live system.

**What makes it annoying**: Alert sounds that cry wolf (sounds urgent, content is mundane). Constant narration of routine updates. Music that is too intense for casual monitoring.

### 6c. News and Current Events Streams

For streams like an Onion-style parody news or serious current events coverage:

**Recommended model**: Classic broadcast news audio design.
- Opening theme sting (branded, 8-10 seconds)
- Low ambient music bed during coverage (barely audible)
- Breaking news stinger when transitioning to urgent story
- Transition bumpers between segments
- Closing theme sting

**Two-voice approach**: This is where the NotebookLM-style anchor + correspondent pattern shines.

```typescript
// Agent generates script with role assignments
const script = `
ANCHOR: Good evening. Three significant developments in our monitoring today.
ANALYST: Starting with the highest urgency item: satellite imagery shows unusual activity near the Bandar Abbas naval base.
ANCHOR: What are we seeing specifically?
ANALYST: Approximately 12 vessels have repositioned over the last 18 hours. That's outside normal patrol patterns.
ANCHOR: The significance?
ANALYST: Taken with the diplomatic communications we've been tracking, it suggests preparation rather than exercise activity.
`;

// Parse and queue each line to appropriate TTS voice
parseScript(script).forEach(({ role, text }) => {
  const voiceId = role === 'ANCHOR' ? 'nova' : 'onyx';  // OpenAI voices
  audioQueue.add({ text, voiceId, priority: 'normal' });
});
```

**What makes it engaging**: News format is familiar and trusted. Two voices create the sense of a live broadcast, not a single robotic reader. The back-and-forth suggests analysis and depth.

**What makes it annoying**: If the agent generates bad dialogue (repetitive, unnatural, too formal). If the pacing is wrong (one voice talking for too long without the other). If the content does not justify the news format.

### 6d. Financial / Market Streams

**Ambient layer:**
- Subtle market ambient (jazz-influenced, professional, Bloomberg-style)
- Soft ticking sound (like a trading terminal)
- Price change tones (ascending for positive, descending for negative, proportional to magnitude)

**Narration style:**
- Terse, data-first: "$GME up 3.4%. Volume 2.3x average. Options flow suggesting..."
- No editorializing unless the data itself is dramatic
- Regular structured summary intervals (every 5 minutes: top movers, sector summary)

**What makes it engaging**: Financial data has inherent narrative -- prices go up or down for reasons, and tracking those reasons is compelling. Audio that reflects the rhythm of the market (quiet during off-hours, active during market hours) creates authenticity.

### 6e. What Audio Patterns Keep Viewers Engaged vs Drive Them Away

**Engagement patterns:**
1. **Consistent sonic identity**: The stream has a recognizable sound. Viewers identify it like they identify a radio station.
2. **Audio hierarchy**: Quiet background, medium events, loud only for truly important moments. The hierarchy teaches viewers to listen for cues.
3. **Appropriate silence**: Not every moment needs sound. Well-placed silence is dramatic.
4. **Voice that matches content**: Urgent narration for urgent content, calm for routine. Never calm narration for breaking news.
5. **Rewarding attention**: Sound design that reveals information to attentive listeners (subtle data tones that track what the agent is monitoring).

**Driving viewers away:**
1. **Alert fatigue**: Sounding urgent too often. After 10 minutes of "breaking news" sounds for routine updates, viewers mute.
2. **Loud background music**: The music should never compete with the narration or make the stream tiring.
3. **Robotic narration**: Even at 10% density, if the voice sounds like a 2010-era TTS system, it undermines the content.
4. **Unsynchronized audio**: TTS audio describing something that is not yet on screen, or describing something that was on screen 3 seconds ago.
5. **Repetitive sounds**: The same chime every 30 seconds for 5 hours becomes excruciating.
6. **No audio at all**: Silence for extended periods without intentional design signals that something is broken.

## 7. Audio Architecture Recommendation

### 7a. Recommended Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Primary TTS | OpenAI `gpt-4o-mini-tts` | Instruction-following, good quality, $0.054/hr at 10% |
| Branded voice | ElevenLabs (pre-generated, cached) | Quality for intro/outro only |
| Fallback TTS | Piper TTS (self-hosted) | Zero cost if API budget exhausted |
| Background music | Pre-generated ACE-Step/Suno library | Zero marginal cost |
| Sound effects | Pre-generated + Web Audio API synthesis | Zero marginal cost |
| Audio mixing | Web Audio API in React | Runs in Chrome, captured via PulseAudio |
| Capture | PulseAudio virtual sink + FFmpeg | Standard headless Chrome audio capture |

### 7b. Audio Event Types in Agent Contract

Audio should be a first-class concept in the agent-renderer contract:

```typescript
// Types for audio events in the stream contract
type AudioEvent =
  | {
      type: 'narrate';
      text: string;
      voice?: 'anchor' | 'analyst' | 'correspondent';
      instructions?: string;    // OpenAI tone instructions
      priority?: 'normal' | 'urgent';
    }
  | {
      type: 'music';
      action: 'set' | 'fade-to' | 'stop';
      track?: 'calm' | 'moderate' | 'urgent' | 'data' | 'tech' | 'weather';
      volume?: number;          // 0-1, default 0.07
    }
  | {
      type: 'sfx';
      sound: 'chime-soft' | 'chime-medium' | 'alert-urgent' | 'transition' | 'resolved' | 'error';
    }
  | {
      type: 'silence';
      duration: number;         // ms; intentional pause
    };

// Example: agent encounters breaking development
const breakingNewsAudio: AudioEvent[] = [
  { type: 'sfx', sound: 'alert-urgent' },
  { type: 'music', action: 'fade-to', track: 'urgent', volume: 0.04 },
  {
    type: 'narrate',
    text: 'Breaking development: new sanctions announced targeting Iranian crude oil exports.',
    voice: 'anchor',
    instructions: 'Speak with gravitas and urgency, like a senior news anchor.',
    priority: 'urgent',
  },
];

// Example: routine data update
const routineUpdateAudio: AudioEvent[] = [
  { type: 'sfx', sound: 'chime-soft' },
  // No narration for routine updates
];

// Example: topic transition
const topicTransitionAudio: AudioEvent[] = [
  { type: 'sfx', sound: 'transition' },
  { type: 'music', action: 'fade-to', track: 'calm' },
  {
    type: 'narrate',
    text: 'Shifting focus now to the Pacific trade situation.',
    voice: 'anchor',
    instructions: 'Measured, transitional tone.',
  },
];
```

```

  |
  | Generates AudioEvent[] alongside visual content updates
  | Audio events can be: narrate / music / sfx / silence
  |
  v
Stream Backend (Node.js)
  |
  | Receives audio events from agent
  | For 'narrate' events: calls OpenAI TTS API, streams PCM audio
  | Selects music track and SFX from pre-generated library

  v
WebSocket to React App (in Chrome Sandbox)
  |

  | Music/SFX file paths referenced for browser-side playback
  |
  v

  |-- AudioContext (44100Hz)
  |   |-- TTS AudioWorklet (streaming PCM playback)
  |   |    +-- TTS GainNode (1.0, full volume)
  |   |-- Music MediaElementSource
  |   |    +-- Music GainNode (0.07, with auto-ducking during TTS)
  |   |-- SFX MediaElementSource (triggered on event)

  |
  v

  |
  v

  |
  v

  |
  v
Twitch / YouTube / dazzle.fm
```

## 8. Open Questions

1. **Per-channel audio personality**: Should weather monitoring and coding streams sound distinctly different? The contract supports this, but it requires maintaining separate music/SFX libraries per stream type, or at minimum a categorization of existing assets.

2. **Viewer audio controls on dazzle.fm**: Should viewers be able to independently control narration volume vs. music volume vs. SFX volume? This requires sending separate audio channels, which is complex with RTMP. Possible on the custom dazzle.fm player (different audio tracks in the stream) but not on Twitch/YouTube. Decision: custom player could offer per-layer control; Twitch/YouTube get a single mixed track.

3. **Non-English narration**: OpenAI `gpt-4o-mini-tts` and Google TTS support many languages. Should Dazzle streams be multilingual from the start? The architecture supports it (just pass different language instructions), but SFX and music design may need to account for non-Western audiences.

4. **First person vs. third person narration**: "I've identified a new development" vs. "The agent has identified a new development." First person is more engaging (Twitch streamer model) and treats the agent as a character. Third person is more journalistic. This should probably be a per-stream/per-agent setting with a default of first person for Razzle.

5. **OpenAI Realtime API for viewer interaction**: The Realtime API enables two-way voice: viewers could speak to the agent and the agent responds. This is a significant future feature (live voice Q&A during streams). Current architecture (PulseAudio capture of all Chrome audio) would need modification to handle microphone input. Worth noting as a roadmap item.

6. **Dynamic music generation**: If ACE-Step generation speed improves to near-real-time, we could generate music that matches the stream's current emotional state or content. For example, when breaking news hits, generate an urgency-appropriate track rather than pulling from a preset library. Not currently feasible at the generation speeds needed, but a notable future direction.

7. **Audio fingerprinting / copyright risk**: Pre-generated AI music (from Suno, Udio) might theoretically be flagged by YouTube's Content ID system if it sounds similar to existing music (a known issue with some AI-generated tracks). ACE-Step (open-source, self-generated) carries lower risk. Worth testing Suno/Udio output against Content ID before using on YouTube-published streams.

### Cost Calculator

For `gpt-4o-mini-tts` at $12.00 per 1M characters:

```
cost_per_hour = (narration_density_pct / 100) * 60 * 150 * 5 * (12.00 / 1_000_000)
```

At 10%: `(0.10) * 60 * 150 * 5 * 0.000012 = $0.054/hr`

At 15%: `(0.15) * 60 * 150 * 5 * 0.000012 = $0.081/hr`

**Practical budget target**: Keep narration density at 10-15% to stay under $0.10/hr for TTS, keeping total stream cost (compute + audio) around $0.15-0.20/hr.
